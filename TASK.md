# Техническое задание: Система бронирования переговорок «SlotMaster»

## 1. Общее описание проекта
**SlotMaster** — это внутренний корпоративный сервис для автоматизации бронирования переговорных комнат. Система позволяет администраторам гибко настраивать графики доступности помещений, а сотрудникам — бронировать фиксированные 30-минутные интервалы (слоты) без конфликтов и пересечений.

### Стек технологий
* **Backend:** Go 1.21+ (чистая архитектура / Layered Architecture)
* **Database:** PostgreSQL 15+
* **Frontend:** React 18, TypeScript, Vite
* **API:** REST (JSON over HTTP)
* **Auth:** JWT (JSON Web Token)

---

## 2. Модель данных (Entity Relationship)

В системе используется 5 основных таблиц для обеспечения бизнес-логики:

1.  **`users`**: Пользователи системы.
    * `id` (UUID, PK): Уникальный идентификатор.
    * `email` (VARCHAR, Unique): Почта сотрудника.
    * `role` (ENUM): 'admin' или 'user'.
    * `created_at` (TIMESTAMP): Дата регистрации.

2.  **`rooms`**: Реестр переговорных комнат.
    * `id` (UUID, PK): Уникальный идентификатор.
    * `name` (VARCHAR): Название (напр., «Альфа», «Стеклянная»).
    * `description` (TEXT): Описание оборудования.
    * `capacity` (INT): Вместимость человек.

3.  **`schedules`**: Правила доступности комнат.
    * `id` (UUID, PK).
    * `room_id` (FK -> rooms.id).
    * `day_of_week` (INT): 0 (вс) – 6 (сб).
    * `start_time` (TIME): Время открытия окна (напр., 09:00:00).
    * `end_time` (TIME): Время закрытия окна (напр., 18:30:00).

4.  **`slots`**: Конкретные временные интервалы в календаре.
    * `id` (UUID, PK).
    * `room_id` (FK -> rooms.id).
    * `start_at` (TIMESTAMP UTC): Точное начало (напр., 2026-04-13 10:00:00).
    * `end_at` (TIMESTAMP UTC): Точное окончание (напр., 2026-04-13 10:30:00).
    * *Constraint:* Уникальный индекс (room_id, start_at).

5.  **`bookings`**: Активные и отмененные брони.
    * `id` (UUID, PK).
    * `user_id` (FK -> users.id).
    * `slot_id` (FK -> slots.id, Unique): Один слот — одна бронь.
    * `status` (ENUM): 'active', 'cancelled'.
    * `created_at` (TIMESTAMP UTC).

## 3. Функциональные требования

### 3.1. Роль: Администратор (Admin)
* **Управление комнатами:** Создание новых переговорок. Редактирование и удаление недоступны в рамках MVP.
* **Настройка расписания:** Однократное создание правил доступности для комнаты (выбор дней и времени). 
    * *Бизнес-правило:* Расписание нельзя изменить после создания.
* **Глобальный мониторинг:** Просмотр списка всех бронирований всех пользователей с пагинацией.

### 3.2. Роль: Пользователь (User)
* **Поиск:** Просмотр списка всех переговорок.
* **Бронирование:** Просмотр доступных слотов на выбранную дату и бронирование свободного слота. В случае бронирования слота с прошедшим временем возврашать 400
* **Управление своими бронями:** Просмотр списка своих **будущих** бронирований и возможность их отмены.
    * *Бизнес-правило:* Пользователь не может бронировать слоты в прошлом.
    * *Бизнес-правило:* Отмена брони идемпотентна.

### 3.3. Системные процессы
* **Генерация слотов:** Система автоматически нарезает доступное время из расписания на 30-минутные отрезки. 
* **Таймзоны:** Все операции на бэкенде и в БД выполняются строго в **UTC**. Конвертация в локальное время происходит только на стороне клиента.

## 4. Архитектура API

| Метод  | Эндпоинт | Описание | Доступ |
| :--- | :--- | :--- | :--- |
| POST | `/dummyLogin` | Получение тестового JWT токена | Public |
| GET | `/rooms` | Получить список всех переговорок | Auth |
| POST | `/admin/rooms` | Создать новую переговорку | Admin |
| POST | `/admin/schedules` | Создать расписание для комнаты | Admin |
| GET | `/slots` | Получить слоты (фильтры: room_id, date) | Auth |
| POST | `/bookings` | Забронировать конкретный slot_id | User |
| GET | `/bookings/my` | Список будущих броней текущего юзера | User |
| DELETE | `/bookings/{id}` | Отменить бронь | User (owner) |
| GET | `/admin/bookings` | Все брони в системе (пагинация) | Admin |

## 5. Требования к интерфейсу (React)

1.  **Auth Screen:** Выбор между "Войти как Админ" и "Войти как Юзер" (для теста).
2.  **Room List:** Сетка карточек с названием и вместимостью комнат.
3.  **Booking Dashboard:**
    * Календарь (Date Picker).
    * Интерактивная таблица слотов (Slot Grid), где занятые слоты окрашены в серый цвет, а доступные — в зеленый.
4.  **User Dashboard:** Список карточек предстоящих встреч с кнопкой «Отменить».
5.  **Admin Panel:** * Форма создания комнаты.
    * Сложная форма расписания (мультивыбор дней недели: Пн-Пт).
    * Таблица всех бронирований системы.

## 6. Нефункциональные требования
* **Безопасность:** Проверка JWT в Middleware на стороне Go.
* **Производительность:** Генерация слотов должна быть оптимизирована (индексы в БД).
* **Надежность:** Использование SQL-транзакций при создании записи в `bookings`, чтобы избежать Race Condition (двойного бронирования одного слота).
